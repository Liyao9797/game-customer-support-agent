# 飞书协同 Workflow

## 1. 目标

飞书协同用于把玩家反馈从“聊天记录”转成团队可处理、可追踪、可复盘的工作流。MVP 阶段优先接入或预留以下能力：

- 反馈写入飞书文档或多维表格。
- P0/P1 飞书群告警。
- 告警去重与聚合。
- 每日玩家反馈简报。
- 处理状态回写。

## 2. 飞书分发规则

| 优先级 | 飞书动作 | 通知对象 |
| --- | --- | --- |
| P0 | 群聊 @所有人，发送告警卡片 | 客服、运营、策划、技术、测试 |
| P1 | @对应负责人或值班同学 | 分类默认负责人 |
| P2 | 写入反馈文档/多维表格，进入每日简报 | 相关团队异步查看 |
| P3 | 仅入库，用于聚类和知识库优化 | 暂不主动通知 |

## 3. 反馈表字段

建议飞书多维表格字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| feedback_id | 文本 | 反馈编号 |
| project | 单选 | 项目名称 |
| created_at | 日期时间 | 创建时间 |
| status | 单选 | 处理状态 |
| owner | 人员 | 当前负责人 |
| uid | 文本 | 玩家 UID |
| channel | 单选 | 渠道 |
| login_version | 文本 | 当前版本 |
| device | 文本 | 设备 |
| os | 文本 | 系统 |
| category | 单选 | 一级分类 |
| sub_category | 单选 | 二级分类 |
| priority | 单选 | P0/P1/P2/P3 |
| emotion | 单选 | 情绪 |
| risk_flags | 多选 | 风险标签 |
| ai_summary | 多行文本 | AI 摘要 |
| raw_question | 多行文本 | 原始问题 |
| entities | 多行文本 | 关键信息 JSON |
| duplicate_group_id | 文本 | 聚合问题 ID |
| related_feedback_count | 数字 | 同类反馈数 |
| handling_note | 多行文本 | 处理记录 |
| resolution | 多行文本 | 处理结论 |
| closed_at | 日期时间 | 关闭时间 |

## 4. P0 告警卡片

```text
【P0 玩家反馈告警】

项目：示例棋牌项目
分类：支付充值 / 充值成功但道具未到账
反馈数量：37
涉及版本：1.5.3
涉及渠道：App Store
情绪：明显负面
风险标签：payment, complaint

问题摘要：
多名玩家反馈充值后欢乐豆未到账，存在支付链路或到账延迟风险。

代表反馈：
- FB202605250001：我充值了 30 元没到账
- FB202605250008：扣钱了但是豆子没给

建议负责人：支付/客服
当前状态：submitted
```

P0 发送规则：

- 群聊 @所有人。
- 同类问题 30 分钟内只发送一条首告警。
- 若同类反馈继续快速上涨，发送升级告警。
- 卡片需包含反馈表链接或聚合详情链接。

## 5. P1 告警卡片

```text
【P1 玩家反馈提醒】

项目：示例棋牌项目
分类：活动奖励 / 奖励未到账
反馈数量：6
涉及活动：连续挑战活动
问题摘要：
多名玩家反馈完成活动任务后奖励未到账，需要活动运营确认配置和发奖链路。

建议负责人：活动运营
当前状态：triaged
```

P1 发送规则：

- @默认负责人或当日值班同学。
- 同一聚合问题 30 分钟内不重复打扰。
- 状态变更为 processing 或 resolved 时，可更新原卡片或发送处理进展。

## 6. 告警去重与聚合

聚合维度：

```text
project + category + sub_category + login_version
```

窗口期：

```text
30 分钟
```

聚合逻辑：

1. 新反馈入库后，计算 duplicate_group_key。
2. 查询 30 分钟内是否已有相同 group。
3. 若无 group，创建 duplicate_group_id。
4. 若已有 group，更新 related_feedback_count。
5. 若已推送告警，不重复推送，只更新聚合计数。
6. 若计数达到升级阈值，发送升级告警。

## 7. 同步策略

### 7.1 实时同步

适用：

- P0。
- P1。
- 玩家明确提交的人工反馈。

动作：

- 立即写入飞书反馈表。
- P0/P1 同步触发群通知。
- 写入同步状态和通知状态。

### 7.2 定时同步

适用：

- P2。
- P3。
- 规则咨询沉淀。
- 低频建议。

动作：

- 每 5-15 分钟批量写入飞书表。
- 失败时记录错误原因并重试。

## 8. 失败重试

建议字段：

- sync_status：pending / success / failed / retrying。
- sync_retry_count：重试次数。
- sync_error_message：最近一次错误。
- last_sync_at：最近同步时间。

重试规则：

- 第 1 次失败：1 分钟后重试。
- 第 2 次失败：5 分钟后重试。
- 第 3 次失败：15 分钟后重试。
- 连续失败后写入本地失败队列，等待人工检查配置。

## 9. 每日简报

发送时间建议：

```text
每天 18:00
```

简报结构：

```text
【每日玩家反馈简报】

一、今日概览
- 反馈总量：
- AI 自动解决率：
- 转人工率：
- P0/P1 数量：
- 负面情绪占比：

二、高优问题
- 问题摘要：
- 反馈数量：
- 涉及版本/渠道/设备：
- 当前状态：
- 建议负责人：

三、高频诉求 Top 5
- 按反馈量、情绪、影响链路综合排序

四、AI 产品建议
- 可直接转需求池的优化建议
- 可加入知识库的 FAQ 建议

五、待跟进事项
- 未关闭 P0/P1：
- 长时间未处理问题：
- 需要策划/运营确认的事项：
```

## 10. 接口预留

MVP 可先使用 mock adapter，后续替换真实飞书 API。

建议抽象：

```text
FeishuFeedbackAdapter
- syncFeedback(feedback)
- updateFeedbackStatus(feedback_id, status, handling_note)
- sendAlert(alertPayload)
- sendDailyReport(reportPayload)
- syncKnowledgeItem(knowledgeItem)
```

## 11. 权限与安全边界

- 不在群告警中展示完整手机号、实名信息、身份证等敏感内容。
- 订单截图、支付凭证等敏感附件只在反馈表中保留链接或内部可见附件。
- 账号找回、处罚申诉、未成年人支付必须转人工处理。
- Agent 不在飞书中生成赔付承诺，只输出“建议核查/建议跟进”。
