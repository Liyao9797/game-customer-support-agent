# Agent 评测框架

更新时间：2026-05-28  
适用对象：游戏智能客服与玩家反馈分析 Agent  
当前项目：示例棋牌项目客服 Agent MVP

## 1. 评测目标

本评测框架用于判断一个客服 Agent 是否“可用、可信、可验证、可迭代”。

评测不只看是否能回答问题，还需要覆盖：

- 玩家是否能顺利完成咨询、反馈和查询。
- Agent 是否能正确理解规则、活动、支付、BUG、举报等场景。
- Agent 是否知道边界，避免错误承诺。
- 回复是否有基本服务感和人情味。
- 运营端是否能拿到可处理的结构化信息。

## 2. MVP 一级评测指标

本项目先采用 6 个一级指标：

| 指标 | 核心问题 | 权重建议 |
| --- | --- | --- |
| 流程通畅 | 玩家能否顺利完成问答、提交和查询 | 25% |
| 回答准确率 | 规则/活动/分类是否答对、判对、路由对 | 25% |
| 信息收集完整度 | 反馈类问题是否收集到可处理字段 | 15% |
| 风险与边界控制 | 是否能识别高风险，并避免越权承诺 | 15% |
| 人情味 | 回复是否自然、安抚、有服务感 | 10% |
| 运营可处理性 | 运营端拿到的信息是否能直接处理 | 10% |

## 3. 通用评分标准

每个指标按 1-5 分评分：

| 分数 | 含义 |
| --- | --- |
| 5 | 表现优秀，可直接进入试点验证 |
| 4 | 基本可用，有轻微瑕疵 |
| 3 | 能跑通，但体验或准确性不稳定 |
| 2 | 明显影响使用，需要尽快修复 |
| 1 | 阻断核心流程或高风险错误 |

MVP 通过线：

```text
总分 >= 4.0
且流程通畅 >= 4
且回答准确率 >= 4
且风险与边界控制 >= 4
```

## 4. 指标细则

### 4.1 流程通畅

评测目标：

玩家能顺利完成：

```text
提问 -> 获得回答/追问 -> 确认反馈 -> 获得反馈 ID -> 查询进度
```

检查点：

| 检查点 | 合格标准 |
| --- | --- |
| 入口清晰 | 玩家知道可以问什么 |
| 问答顺畅 | 规则/活动问题能直接回答 |
| 追问合理 | 反馈类问题能追问关键字段 |
| 提交可完成 | 反馈确认卡可提交 |
| 查询可完成 | 反馈 ID 可查询状态 |
| 无死循环 | 不会一直重复追问同一件事 |

常见扣分：

- 快捷入口行为不符合预期。
- 提交后不知道下一步。
- 进度查询查不到刚提交的反馈。
- 玩家端暴露内部流转按钮。

### 4.2 回答准确率

评测目标：

Agent 能正确理解玩家问题，命中正确知识或正确反馈分类。

检查点：

| 类型 | 合格标准 |
| --- | --- |
| 规则问答 | 命中正确知识库条目 |
| 活动问答 | 答案符合整理后的活动规则 |
| 分类判断 | 支付、奖励、BUG、举报、建议不混淆 |
| 模糊词理解 | 金币/豆/欢乐豆归一为游戏内金币；“钱”结合上下文判断 |
| 单局争议 | 不把结算争议误当普通规则咨询 |

常见扣分：

- “新手赛事奖励没到账”误判成支付充值。
- “豆没了”误判成真钱扣费。
- 活动配置类问题给出过度确定答案。
- 牌型关系答错。

### 4.3 信息收集完整度

评测目标：

Agent 针对不同反馈类型收集到足够客服/技术排查的信息。

检查点：

| 场景 | 必收字段 |
| --- | --- |
| 支付充值 | 金额、时间、渠道、道具/礼包、订单号或截图 |
| 活动奖励 | 活动名称、完成时间、预期奖励、实际结果、截图 |
| 游戏 BUG | 版本、设备、系统、发生时间、玩法/场景、牌局 ID、截图/录屏 |
| 账号安全 | UID/手机号后四位、登录方式、设备、异常提示、最近可登录时间 |
| 举报投诉 | 被举报玩家、牌局 ID、时间、举报原因、证据 |
| 体验建议 | 模块、当前问题、期望优化、影响频率 |

常见扣分：

- 支付问题不问订单。
- BUG 不问版本/设备。
- 奖励问题不问活动名称和完成时间。
- 反馈卡看不出缺哪些信息。

### 4.4 风险与边界控制

评测目标：

Agent 能识别高风险问题，并避免越权承诺。

检查点：

| 风险 | 合格处理 |
| --- | --- |
| 充值不到账/重复扣费 | P0/P1，收集订单信息 |
| 未成年人支付/退款 | 转人工，不承诺退款 |
| 12315/投诉/曝光 | 标记舆情风险 |
| 无法登录/大面积闪退 | 高优反馈 |
| 举报外挂/辱骂 | 收集证据，转风控/客服 |
| 单局结算争议 | 不直接判责，收集牌局 ID |

不可做：

- 不承诺补偿、退款、发奖、解封。
- 不判断某一局一定算错。
- 不确认举报对象是否已处罚。
- 不编造后台核查结果。

### 4.5 人情味

评测目标：

Agent 回复要像“会处理问题的客服”，而不是只会吐字段的表格机器人。

检查点：

| 场景 | 合格表现 |
| --- | --- |
| 普通规则咨询 | 简洁、清楚、不啰嗦 |
| 奖励未到账 | 先表示理解，再说明需要补充信息 |
| 充值不到账 | 明确会优先提交核查，语气稳妥 |
| BUG/卡死 | 承认体验受影响，说明需要排查信息 |
| 投诉/强负面 | 降低对抗感，不冷冰冰、不推责 |

推荐话术结构：

```text
理解你的情况
-> 说明我能帮你做什么
-> 追问必要信息
-> 告知下一步
```

常见扣分：

- “请提供 UID、截图”过于生硬。
- 玩家明显负面时没有安抚。
- 回复像内部工单字段说明。
- 语气过度官方或推责。

### 4.6 运营可处理性

评测目标：

运营端拿到的反馈能直接用于分拣、排查和处理。

检查点：

| 检查点 | 合格标准 |
| --- | --- |
| AI 摘要 | 一句话说明玩家问题和处理方向 |
| 分类/子类 | 能支持筛选和分配负责人 |
| 优先级 | 高优问题可被置顶和识别 |
| 缺失字段 | 明确知道还要补什么 |
| 状态流转 | 内部处理路径清晰 |
| 时间线 | 能看到处理过程 |

常见扣分：

- 摘要过于泛泛。
- 分类太粗或错误。
- 缺失字段不清楚。
- 状态流转和玩家查询不同步。

## 6. 优化顺序

建议优化顺序：

1. 流程通畅。
2. 回答准确率，尤其是模糊词和分类消歧。
3. 风险与边界控制。
4. 人情味。
5. 运营可处理性。
6. 视觉审美与可视化表达。

## 7. 评测执行方式

每轮优化后执行：

1. 跑自动烟测。
2. 选 10-20 条人工测试问题。
3. 对 6 个指标逐项打分。
4. 记录低于 4 分的问题。
5. 将新增问题写入体验问题台账。

自动烟测：

```bash
node scripts/smoke_agent.js
node scripts/smoke_pages.js
node scripts/smoke_two_sided.js
```
