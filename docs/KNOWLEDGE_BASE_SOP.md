# 知识库更新 SOP

## 1. 目标

将游戏规则、玩法说明、活动规则、客服常见问题等资料整理为 Agent 可检索、可回答、可复用的标准知识条目。MVP 阶段先支持文字和截图输入，后续可接入飞书文档自动同步。

## 2. 输入来源

- 游戏内规则截图。
- 活动页面截图。
- 产品同学提供的规则文字。
- 客服 FAQ 文档。
- 玩家高频问题日报。
- 策划或运营补充说明。

## 3. 处理流程

```text
产品同学提供文字或截图
-> AI 提取规则信息
-> 转换为标准知识条目
-> 生成常见问法和标准回答
-> 标注澄清策略
-> 标注不可回答边界
-> 人工确认
-> 同步飞书规则文档
-> 更新 Agent 检索库
-> 用测试问题验证回答
```

## 4. 标准知识条目字段

| 字段 | 说明 |
| --- | --- |
| knowledge_id | 知识 ID |
| project | 项目 |
| module | 模块 |
| topic | 规则主题 |
| source | 来源文字/截图 |
| source_status | 资料完整性 |
| common_questions | 常见问法 |
| answer | 标准回答 |
| clarification_strategy | 澄清策略 |
| no_answer_boundary | 不可回答边界 |
| tags | 检索标签 |
| updated_at | 更新时间 |

## 5. 知识 ID 规则

格式：

```text
KB + 项目缩写 + YYYYMMDD + 4 位序号
```

示例：

```text
KB-CARD-20260525-0001
```

## 6. 条目模板

```json
{
  "knowledge_id": "KB-CARD-20260525-0001",
  "project": "示例棋牌项目",
  "module": "玩法规则",
  "topic": "黄金牌活动炸弹计算规则",
  "source": "游戏内规则截图/产品提供文字",
  "source_status": "完整",
  "common_questions": [
    "黄金牌活动炸弹怎么算？",
    "暗牌炸弹倍率是多少？",
    "黄金牌活动和普通炸弹有什么区别？"
  ],
  "answer": "根据当前规则资料，黄金牌活动炸弹会按游戏内牌型和倍率规则结算。具体倍数可能受玩法模式、活动配置和房间规则影响，请以当前对局内规则说明为准。",
  "clarification_strategy": "如果玩家未说明玩法模式或房间类型，先追问具体模式；如果涉及结算争议，引导提交牌局 ID。",
  "no_answer_boundary": "不承诺具体补偿；不判断单局结算是否异常；结算争议需提交牌局 ID 由人工核查。",
  "tags": ["黄金牌活动", "炸弹", "倍率", "结算", "玩法规则"],
  "updated_at": "2026-05-25 10:30:00"
}
```

## 7. 资料完整性标记

| source_status | 含义 | 处理方式 |
| --- | --- | --- |
| 完整 | 规则信息充分，可直接回答 | 进入检索库 |
| 部分缺失 | 缺少模式、版本、限制条件等信息 | 回答时加入边界，等待补充 |
| 待确认 | OCR 不清晰或规则可能过期 | 不直接用于确定性回答 |
| 已过期 | 活动结束或版本不适用 | 从默认检索结果中下线 |

## 8. 回答规范

标准回答应满足：

- 先直接回答玩家问题。
- 用玩家能理解的语言，不照搬复杂规则。
- 对版本、活动、房间差异加边界说明。
- 对不确定信息明确说明“暂无法确认”。
- 不承诺补偿、退款、发奖、解封等处理结果。
- 涉及争议时引导提交反馈并补充必要信息。

## 9. 常见问法生成规则

每条知识至少生成 3-5 个常见问法：

- 玩家口语表达。
- 规则名称表达。
- 错别字或简称表达。
- 问结算、倍率、奖励、限制条件的表达。
- 与相邻规则混淆的表达。

示例：

```text
鱼票会清空吗？
连续挑战活动结束后鱼票还在吗？
活动没用完的鱼票怎么办？
钓鱼活动道具会过期吗？
```

## 10. 澄清策略

当玩家问题缺少关键信息时，Agent 不应猜测，应优先追问。

| 模块 | 需要澄清的信息 |
| --- | --- |
| 玩法规则 | 玩法模式、房间类型、牌局 ID |
| 活动规则 | 活动名称、活动页面、时间范围 |
| 奖励问题 | 活动名称、完成时间、预期奖励 |
| 结算争议 | 牌局 ID、截图、发生时间 |
| 账号问题 | 登录方式、设备、异常提示 |

## 11. 不可回答边界

以下问题不能由知识库直接给最终结论：

- 某一单充值是否应补发。
- 某一账号是否能解封。
- 某一局牌是否结算错误。
- 某一举报对象是否已被处罚。
- 退款、赔付、补偿承诺。
- 需要后台数据、订单、账号或风控系统核查的问题。

标准引导：

```text
这个问题需要结合后台记录核查。我可以先帮你提交反馈，请补充相关信息，客服同学会继续处理。
```

## 12. 飞书规则文档结构

建议按模块拆分：

- 玩法规则。
- 活动规则。
- 充值支付 FAQ。
- 账号安全 FAQ。
- 举报投诉 FAQ。
- BUG 反馈指引。
- 体验建议收集指引。

每个模块包含：

- 知识 ID。
- 主题。
- 标准回答。
- 常见问法。
- 不可回答边界。
- 更新时间。
- 负责人。

## 13. 更新频率

- 活动规则：每次活动上线或配置变更后更新。
- 玩法规则：版本更新或玩法改动后更新。
- 支付/账号 FAQ：客服策略变更后更新。
- 高频问题：每日简报后评估是否补充。

## 14. 验证用例

每次更新知识库后至少验证：

1. 标准问法能命中正确知识。
2. 口语问法能命中正确知识。
3. 相似但不同的问题不会误答。
4. 缺少关键信息时会追问。
5. 超出边界的问题会引导提交反馈。

## 15. 复用到其他游戏项目

迁移到新项目时需要替换：

- project。
- 项目缩写。
- 模块清单。
- 分类体系中的默认负责人。
- 玩法规则和活动规则资料。
- 飞书文档和群配置。

可复用内容：

- 知识条目字段。
- 常见问法生成规则。
- 不可回答边界。
- 澄清策略。
- 验证流程。
